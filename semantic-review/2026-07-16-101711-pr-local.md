# Credential-backed access and separated operator views

The change removes the exposed Dev Login/default credential path and replaces it with an email-and-password login backed by a new users table and JWTs. An initial administrator can now be created only from explicitly supplied bootstrap settings, while user creation is guarded by an admin-token check. The frontend reorganizes the former single Dashboard into an Overview, Operations, and AI Intelligence route, and restores a persisted session before rendering the control plane. The dashboard split is a useful reduction in the amount of operational detail presented at the landing view.  
**Watch for:** **confirmed** a fixed JWT signing secret is accepted in every environment; **confirmed** the new JWT identity is not connected to the tenant-controlled API; **confirmed** a bootstrap administrator has no tenant context and falls into the standalone workspace; **confirmed** admin tokens retain privilege after a role change, disablement, or deletion; and **confirmed** the registration client cannot authenticate its own admin-only request.

## High-level view

Bootstrap configuration is opt-in and validates that email, username, and password are supplied together, which removes a seeded account from the checked-in configuration. The resulting JWT is nevertheless signed with a fixed fallback in staging and production unless each deployment overrides it. **Confirmed:** the fallback makes any token payload, including an `admin` role, forgeable by an attacker who knows the source value.

The authentication module has its own JWT check, while the existing control-plane routes continue to derive the organization solely from `X-Organization-Id`. The frontend follows the same split: it verifies its token only at `/auth/verify-token` and sends no bearer token on normal data requests. **Confirmed:** this leaves a successful login unable to establish a tenant identity for the application, and deployed environments still fail closed because the old development tenant adapter is intentionally unavailable there.

The bootstrap user is intentionally global, but the frontend needs an organization ID to initialize its workspace. It substitutes the user UUID when no organization is present and immediately tries to fetch that nonexistent organization. **Confirmed:** the default explicit-bootstrap path lands in the standalone fallback instead of an initialized dashboard.

Overview now contains KPIs and action links, with validation workflows isolated under Operations and AI Proof-of-Risk content isolated under AI Intelligence. The new roles are only enforced for the auth router; an issued admin token keeps its claim for 24 hours even if the underlying user is later changed or removed. **Confirmed:** the same token remains accepted by registration and user-management endpoints.

## JWT signing falls back to a public, cross-environment secret

`Settings.jwt_secret` was added with the literal `securescope-dev-jwt-secret-change-in-production-2026` as a default (`backend/app/config.py:59-61`). No validator requires `SECURESCOPE_JWT_SECRET` outside development, while the new auth router accepts a token when PyJWT can verify it with that setting (`backend/app/modules/auth/router.py:48-59`). **Confirmed:** a production deployment that follows the new bootstrap documentation but omits this additional setting accepts attacker-created HS256 tokens carrying an arbitrary subject, organization, and `role: "admin"` claim.

Require a deployment-supplied high-entropy signing secret in staging and production, and reject the known development value there. The bootstrap credentials are secret-wrapped and have all-or-nothing validation, but those protections do not protect the bearer token signing key.

## The JWT login plane is disconnected from tenant-authorized work

The new flow currently has two independent identity paths:

```text
Browser -- Bearer JWT --> /auth/* ------------> JWT role check
Browser -- X-Organization-Id --> /api/v1/* --> development tenant adapter
```

`frontend/src/lib/securescope/authApi.ts` sends the bearer token only to `/auth/me` and `/auth/verify-token`; the established API client in `frontend/src/lib/securescope/api.ts` sends only `X-Organization-Id`. The existing organization and project routers still depend on `require_tenant_context`, which explicitly rejects all staging and production requests because it is a development-only header adapter (`backend/app/modules/tenancy/development_auth.py`). **Confirmed:** logging in establishes neither authorization nor a usable tenant context for core data and operational actions outside development. The header adapter predates this diff; the newly added gap is introducing JWT login without making it the source of tenant identity for those routes.

Derive `TenantContext` from the verified JWT (and enforce its organization membership) for every protected control-plane endpoint, then attach the bearer token from the standard frontend client. Keep the development adapter only as an explicitly isolated local mode.

## Bootstrap administrators cannot initialize an organization workspace

The new settings accept email, username, password, and optional full name, but no organization binding. `ensure_bootstrap_admin_exists` creates the first administrator without `organization_id` (`backend/app/modules/auth/service.py:178-205`). After login, both the direct login path and persisted-session restoration call `login(user.organization_id || user.id)` (`LoginPage.tsx:164` and `page.tsx:36`); `initData` then passes that value as `X-Organization-Id` and requires the organizations query to return a match (`store.ts:212-294`). **Confirmed:** a newly bootstrapped admin uses their user UUID as a tenant UUID, receives no matching organization, and is demoted to the “real scan standalone” fallback instead of the requested dashboard experience.

Define the bootstrap administrator's intended organization relationship, or introduce a post-login tenant-selection/onboarding flow. The server must derive and validate that relationship from the authenticated principal rather than treating a user identifier as an organization identifier.

## Disabled or demoted administrators keep their old token authority

`_require_admin` checks only the `role` embedded in the JWT (`backend/app/modules/auth/router.py:63-69`). The administrator can update a user's `role` or `is_active`, and can delete the account, but none of those actions change a token version, consult the user record, or revoke sessions (`service.py:124-158`). Access tokens are issued for 24 hours (`security.py:51-78`). **Confirmed:** an administrator whose account is disabled, demoted, or deleted may continue to register users and manage users until the token expires.

For privileged endpoints, verify current user state and role from storage, or enforce a revocation/token-version check with a short-lived access-token policy. The same mechanism should invalidate tokens after password changes.

## The admin-only registration client cannot make an authenticated request

`POST /auth/register` correctly requires `_require_admin`, but `registerUser` invokes `authFetch` without an `Authorization` header (`frontend/src/lib/securescope/authApi.ts:60-65`). There is also no registration or user-management page in the changed frontend. **Confirmed:** the client-side registration function receives a 401 if it is ever wired into an administrator workflow, so the new server-side registration capability is not usable from this application.

Pass the current bearer token on administrator actions and add the intended administrator-facing registration path, or remove the unused client helper until that workflow exists.

## Authentication and dashboard behavior are untested

No authentication, bootstrap lifecycle, tenant-binding, role-revocation, login restoration, or dashboard-route tests were added in this local diff. **Confirmed:** failures in the initial administrator's first login and in the role boundary have no automated regression coverage. Not tested: startup with an empty database and explicit bootstrap configuration; deployed-mode JWT-secret validation; admin registration after login; disabling/demoting a user with an existing token; and route rendering for Overview, Operations, and AI Intelligence.

Add backend integration tests around startup and authorization, plus frontend tests that assert token propagation, tenant selection, and each split dashboard route.

## Out-of-scope pentest audit and visual redesign are bundled with this change

**Confirmed:** the same local diff also adds a pentest-audit service and scanner suite, Next.js proxy wiring, a Three.js login background, extensive styling/font work, dependency updates, and a Docker retry change. Those changes are unrelated to credential removal, bootstrap administration, registration, and dashboard separation, so they were not analyzed for their own security or behavioral risks in this review. Splitting them would make the authentication and dashboard reviewable independently.
