# Credential-backed tenant access and split operator views

The change replaces the frontend Dev Login path with credential-backed JWT login, adds an explicit organization-bound bootstrap administrator, and makes `/api/v1` tenant identity come from a verified bearer token before considering the development/test header fallback. Token versions are checked against persisted users, so password, role, and active-state changes revoke existing access. The dashboard now selects separate Overview, Operations, and AI Intelligence views, with the overview limited to readiness and live-safety signals. **Watch for:** **confirmed** an administrator from one organization can register, list, read, modify, or delete users in every organization; **confirmed** the AI and domain-scan paths bypass both the shared bearer client and tenant verification; **confirmed** the new login, bootstrap, revocation, and bearer-precedence behavior has no direct regression coverage.

## High-level view

The standard `/api/v1` client now attaches the stored bearer token, and `require_tenant_context` validates the token signature, persisted user state, organization claim, role, and token version before allowing the development/test header fallback. That boundary is incomplete: the Operations and AI views still invoke direct clients whose requests carry no bearer, while their API routes accept no tenant context.

User administration verifies that the caller is a currently persisted, active administrator, rather than trusting the role claim alone. It does not carry the administrator's organization into the user-management operations, so an administrator can select another organization during registration and enumerate or alter all users globally.

The new `dashboard`, `operations`, and `ai_intelligence` route keys select separate view trees. Overview remains concise—KPIs, a priority queue, and activity—while operations and AI detail are kept out of that landing surface.

## Administrator authority escapes its organization

`POST /auth/register` obtains a current persisted administrator through `_require_admin`, but passes only the unscoped request into `AuthService.register`. The service persists `request.organization_id` verbatim (`backend/app/modules/auth/router.py`, `backend/app/modules/auth/service.py`), so an administrator associated with organization A can create a user in organization B or an unassigned user. The same router exposes `list_users`, `get_user`, `update_user`, and `delete_user` to every current administrator, while the corresponding service queries have no organization predicate. **Confirmed:** the organization binding introduced for bootstrap accounts does not constrain ongoing administration; a tenant administrator can enumerate and change identities belonging to another tenant.

Resolve the administrator record once and pass its persisted organization into every administrative operation. Registration should derive that organization server-side; list and single-user operations should filter by it and return the same not-found result for other tenants.

## Operations and AI requests bypass the new bearer boundary

The shared control-plane client in `frontend/src/lib/securescope/api.ts` attaches `Authorization: Bearer <stored token>`, which protects the ordinary `/api/v1` requests. `frontend/src/lib/securescope/aiProofOfRiskApi.ts` and `frontend/src/lib/securescope/domainSafeScanApi.ts` instead call `fetch` directly with only `Content-Type`. Their matching backend routes, `backend/app/modules/ai_proof_of_risk/router.py` and `backend/app/modules/domain_safe_scan/router.py`, declare neither `require_tenant_context` nor another bearer-verification dependency. The new Operations and AI Intelligence views retain panels that call those clients. **Confirmed:** these paths run without a verified bearer identity or tenant in every environment, so the required global bearer transport and bearer-derived control-plane context do not hold beyond the shared `/api/v1` client.

Make the authenticated request helper reusable by the alternate clients, then require the verified tenant context in both routes and scope the operation to it. Login and token-verification calls remain the deliberate unauthenticated/authentication exceptions.

## Security changes have no direct regression tests

The changed backend test only supplies an explicit JWT secret to the existing environment-guard helper (`backend/tests/integration/test_environment_guards.py`). No test creates the bootstrap user, logs in, asserts current persisted-admin enforcement, changes a password, role, or active flag and reuses the old token, or sends both a bearer and a conflicting tenant header. The frontend tests cover existing dashboard components but do not render `Page`, exercise credential login/session restoration, assert the standard or alternate clients' authorization headers, or check the new route variants. **Confirmed:** the security contract introduced here is not protected against regression. Not tested: deployed rejection of the development signing secret, organization-bound bootstrap behavior, cross-tenant administration denial, token revocation, bearer-over-header precedence, and Overview/Operations/AI Intelligence routing.

Add focused backend integration tests around the authorization boundary and frontend tests around login restoration, request headers, and each split route.

## Unrelated local work remains bundled

**Confirmed:** the local worktree also contains a Pentest Audit feature, its scanner implementation and UI, the Three.js/login visual and broad stylesheet changes, Docker retry tuning, dependency-lock output, and generated TypeScript build metadata. They are unrelated to the requested credential, tenant-context, and dashboard split review and were not assessed for their own behavior or security posture.
